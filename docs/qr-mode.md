# notapipe QR Mode — Air-Gapped WebRTC Deep-Dive

This document covers the technical details of the QR-based connection mode, where two devices establish a WebRTC connection without any server involvement by encoding WebRTC session data into scannable QR codes.

Reference: [Air-gapped WebRTC: Breaking the QR Limit](https://magarcia.io/air-gapped-webrtc-breaking-the-qr-limit/)

---

## The Core Problem

A WebRTC SDP offer is typically 1500–4000 bytes of mostly redundant, human-readable text. A QR Version 40 (maximum size, 177×177 modules) with error correction level L can hold at most **2953 bytes in binary mode**. A raw or even ZLIB-compressed SDP barely fits in the maximum QR size, producing a dense, error-prone code that is difficult to scan reliably.

The solution is **semantic compression**: extract only the meaningful bits from the SDP, encode them in a compact binary format, and reconstruct a complete valid SDP on the receiving side from a template.

## What's Actually in a WebRTC Data-Channel SDP Offer?

```
v=0
o=- 1234567890 2 IN IP4 127.0.0.1
s=-
t=0 0
a=group:BUNDLE 0
a=extmap-allow-mixed
a=msid-semantic: WMS
m=application 9 UDP/DTLS/SCTP webrtc-datachannel
c=IN IP4 0.0.0.0
a=ice-ufrag:AbCd          ← 4 chars, DERIVED from fingerprint
a=ice-pwd:AbCdEfGhIjKlMnOpQrStUvWx  ← 22-24 chars, DERIVED from fingerprint
a=fingerprint:sha-256 AA:BB:CC:DD:...  ← 32 bytes, MUST TRANSMIT
a=setup:actpass
a=mid:0
a=sctp-port:5000
a=max-message-size:262144
a=candidate:1 1 UDP 2122252543 192.168.1.5 54321 typ host  ← MUST TRANSMIT
a=candidate:2 1 UDP 1686052607 203.0.113.1 54321 typ srflx  ← MUST TRANSMIT
```

Most of this is boilerplate. The truly variable data is:

1. **DTLS fingerprint** — 32 bytes (SHA-256)
2. **ICE credentials** (ufrag + password) — can be DERIVED from the fingerprint
3. **ICE candidates** — IP address + port + type for each

Everything else (`v=0`, `m=application`, `a=sctp-port:5000`, etc.) is constant for a data-channel-only connection and can be hardcoded into a template on the receiving side.

## ICE Credential Derivation

The key insight from the QWBP protocol: the ICE ufrag and password don't need to be transmitted — they can be derived deterministically from the fingerprint using HKDF. Both sides run the same derivation, so the credentials are implicitly agreed upon.

```typescript
async function deriveIceCredentials(
  fingerprint: Uint8Array,
): Promise<{ ufrag: string; pwd: string }> {
  const key = await crypto.subtle.importKey("raw", fingerprint, "HKDF", false, ["deriveBits"]);
  const salt = new TextEncoder().encode("notapipe-v1");

  const ufragBits = await crypto.subtle.deriveBits(
    { name: "HKDF", hash: "SHA-256", salt, info: new TextEncoder().encode("ufrag") },
    key,
    32,
  );
  const pwdBits = await crypto.subtle.deriveBits(
    { name: "HKDF", hash: "SHA-256", salt, info: new TextEncoder().encode("pwd") },
    key,
    144, // 18 bytes = 144 bits
  );

  return {
    ufrag: base64url(new Uint8Array(ufragBits)).slice(0, 4),
    pwd: base64url(new Uint8Array(pwdBits)).slice(0, 22),
  };
}
```

**Critical**: The derived credentials must be SET on the `RTCPeerConnection` BEFORE creating the offer:

```typescript
const pc = new RTCPeerConnection(iceConfig);
const { ufrag, pwd } = await deriveIceCredentials(myFingerprint);

// Not directly possible via standard API — see workaround below
```

Wait — there's a chicken-and-egg problem. The fingerprint is generated by the browser as part of creating the `RTCPeerConnection` (it's the DTLS certificate hash). We can't derive credentials before the fingerprint exists, and the fingerprint isn't available until after `setLocalDescription`.

**Workaround**:

1. Create `RTCPeerConnection` and `createDataChannel`
2. Call `createOffer()` and `setLocalDescription()` — this creates the certificate and fingerprint
3. Wait for ICE gathering to complete
4. Extract the fingerprint from `pc.localDescription.sdp`
5. Re-derive ICE credentials from the fingerprint
6. The credentials in the actual SDP will differ from derived ones — **this means we DO have to include ufrag and password in the packet** UNLESS we can force the browser to use our derived credentials

Alternative: Force ICE credentials using `RTCPeerConnection` configuration. The spec allows setting `iceUfrag` and `icePwd` via `RTCPeerConnection.setConfiguration()` in some implementations — but this is NOT widely supported.

**Practical decision**: Include ufrag (4 bytes) and password (22 bytes) in the binary packet. This adds 26 bytes to the payload, bringing total to ~75 bytes — still extremely compact and fits in QR Version 4. Drop the HKDF derivation approach for v1 simplicity. Document HKDF as the v2 optimization.

## Binary Packet Format

```
┌─────────────────────────────────────────────────────────┐
│ Byte 0:    Magic (0x4E = 'N' for notapipe)               │
│ Byte 1:    Version (0x01)                                │
│ Byte 2:    Flags                                         │
│              bit 0: is_answer (0=offer, 1=answer)        │
│              bits 1-7: reserved                          │
│ Bytes 3-34: DTLS SHA-256 fingerprint (32 bytes)          │
│ Byte 35:   ufrag length (uint8)                          │
│ Bytes 36+: ufrag (variable, typically 4 bytes)           │
│ Byte N:    pwd length (uint8)                            │
│ Bytes N+1: pwd (variable, typically 22 bytes)            │
│ Byte M:    candidate count (uint8)                       │
│ Bytes M+1: candidate entries (variable)                  │
└─────────────────────────────────────────────────────────┘
```

### Candidate Entry Format

```
┌────────────────────────────────────────────────────────┐
│ Byte 0: Candidate flags                                 │
│           bits 0-1: address type                        │
│             0b00 = IPv4                                 │
│             0b01 = IPv6                                 │
│             0b10 = mDNS (Chromium privacy mode)         │
│           bit 2: candidate type                         │
│             0 = host                                    │
│             1 = srflx (server-reflexive / STUN-derived) │
│           bits 3-7: reserved                            │
│                                                         │
│ IPv4 (type 0b00):                                       │
│   Bytes 1-4:   IPv4 address (4 bytes)                   │
│   Bytes 5-6:   port (uint16 big-endian)                 │
│   Total: 7 bytes per candidate                          │
│                                                         │
│ IPv6 (type 0b01):                                       │
│   Bytes 1-16:  IPv6 address (16 bytes)                  │
│   Bytes 17-18: port (uint16 big-endian)                 │
│   Total: 19 bytes per candidate                         │
│                                                         │
│ mDNS (type 0b10):                                       │
│   Bytes 1-16:  UUID as raw bytes (16 bytes)             │
│   Bytes 17-18: port (uint16 big-endian)                 │
│   Total: 19 bytes per candidate                         │
└────────────────────────────────────────────────────────┘
```

### Candidate Filtering

Before encoding, filter gathered candidates:

- Include at most 2 **host** candidates (prefer IPv4 over IPv6; include at most 1 of each)
- Include at most 1 **srflx** (server-reflexive) candidate
- Exclude **relay** candidates (TURN — not used in v1)
- If only mDNS candidates exist, include at most 1

Typical result: 2 candidates (1 host IPv4, 1 srflx IPv4) = 14 bytes of candidate data.

### Example Packet Size Calculation

```
Header:          3 bytes
Fingerprint:    32 bytes
ufrag length:    1 byte
ufrag:           4 bytes
pwd length:      1 byte
pwd:            22 bytes
candidate count: 1 byte
2 × IPv4 cands: 14 bytes
────────────────────────
Total:          78 bytes
```

A 78-byte payload in binary QR mode fits comfortably in **QR Version 4** (33×33 modules) with error correction L. This is easily scannable even on a small phone screen.

## SDP Template Reconstruction

On the receiving side, synthesize a complete SDP from a hardcoded template, substituting only the variable fields:

```typescript
function buildSdpFromParts(parts: {
  fingerprint: string; // "AA:BB:CC:..." hex with colons
  ufrag: string;
  pwd: string;
  candidates: string[]; // "candidate:..." lines
  isAnswer: boolean;
}): string {
  const setup = parts.isAnswer ? "active" : "actpass";
  const candidateLines = parts.candidates.map((c) => `a=${c}`).join("\r\n");

  return [
    "v=0",
    "o=- 0 0 IN IP4 127.0.0.1",
    "s=-",
    "t=0 0",
    "a=group:BUNDLE 0",
    "a=extmap-allow-mixed",
    "m=application 9 UDP/DTLS/SCTP webrtc-datachannel",
    "c=IN IP4 0.0.0.0",
    `a=ice-ufrag:${parts.ufrag}`,
    `a=ice-pwd:${parts.pwd}`,
    `a=fingerprint:sha-256 ${parts.fingerprint}`,
    `a=setup:${setup}`,
    "a=mid:0",
    "a=sctp-port:5000",
    "a=max-message-size:262144",
    candidateLines,
    "",
  ].join("\r\n");
}
```

**Important**: The reconstructed SDP must be tested against all target browsers. Chrome, Firefox, and Safari parse SDP slightly differently and may reject malformed templates. Test explicitly with `pc.setRemoteDescription({ type: "offer", sdp: reconstructedSdp })` and verify it doesn't throw.

## QR Code Generation

```typescript
import QRCode from "qrcode";

async function renderSdpQr(canvas: HTMLCanvasElement, packet: Uint8Array): Promise<void> {
  // Convert Uint8Array to a string that QRCode can treat as binary
  const binaryString = Array.from(packet, (byte) => String.fromCharCode(byte)).join("");

  await QRCode.toCanvas(canvas, binaryString, {
    errorCorrectionLevel: "L", // lowest ECC = maximum data capacity
    mode: "byte", // binary mode (not alphanumeric or numeric)
    margin: 2,
  });
}
```

The canvas should be displayed at a minimum of 200×200 px on mobile for reliable scanning. Consider adding a white border (quiet zone) of at least 4 modules.

## QR Code Scanning

### Primary: `BarcodeDetector` API

Available on Chrome 83+, Edge 83+, Chrome Android, Samsung Internet. Zero bundle cost.

```typescript
async function scanWithBarcodeDetector(videoElement: HTMLVideoElement): Promise<Uint8Array> {
  const detector = new BarcodeDetector({ formats: ["qr_code"] });

  return new Promise((resolve, reject) => {
    let animFrameId: number;

    async function scan() {
      try {
        const barcodes = await detector.detect(videoElement);
        if (barcodes.length > 0) {
          const raw = barcodes[0].rawValue;
          // Convert binary string back to Uint8Array
          const bytes = new Uint8Array(raw.length);
          for (let i = 0; i < raw.length; i++) {
            bytes[i] = raw.charCodeAt(i);
          }
          resolve(bytes);
          return;
        }
      } catch (e) {
        // Frame not ready yet, continue
      }
      animFrameId = requestAnimationFrame(scan);
    }

    animFrameId = requestAnimationFrame(scan);
  });
}
```

### Fallback: `zxing-wasm`

Dynamically imported when `BarcodeDetector` is not available:

```typescript
async function getScanner() {
  if (typeof BarcodeDetector !== "undefined") {
    return scanWithBarcodeDetector;
  }
  // Lazy load zxing-wasm fallback
  const { readBarcodeFromImageFile } = await import("zxing-wasm");

```

## Full QR Mode Flow

```
Device A (Offerer)                         Device B (Answerer)
═══════════════════                        ════════════════════
1. Create RTCPeerConnection
2. createDataChannel("yjs")
3. createOffer()
4. setLocalDescription(offer)
5. Wait for ICE gathering complete...
   (show spinner: "Gathering network info")
6. encodeSdpToPacket(localDescription)
7. Render QR to canvas
8. Display: "Show this to the other device"

                                           [User scans QR with Device B camera]

                                           9.  scanQR() → packet bytes
                                           10. decodePacketToSdp(packet) → offerSdp
                                           11. Create RTCPeerConnection
                                           12. setRemoteDescription(offerSdp)
                                           13. createAnswer()
                                           14. setLocalDescription(answer)
                                           15. Wait for ICE gathering complete...
                                           16. encodeSdpToPacket(localDescription)
                                           17. Render QR to canvas
                                           18. Display: "Now show this to the first device"

[User scans QR with Device A camera]

19. scanQR() → packet bytes
20. decodePacketToSdp(packet) → answerSdp
21. setRemoteDescription(answerSdp)

══════════════════ RTCDataChannel opens ══════════════════
22. Yjs provider starts syncing
```

## Known Limitations of QR Mode

1. **Symmetric NAT**: If neither device can reach the other's host or srflx candidates, the connection fails. TURN (v2) solves this. QR mode is most reliable on the same local network, which is the primary use case (laptop → phone in the same room).

2. **ICE gathering time**: Can take 5–30 seconds. The offer QR cannot be displayed until gathering is complete. The answerer should wait visibly.

3. **iOS mDNS candidates**: iOS/Safari may produce only mDNS candidates for the host address. These are resolvable locally but not across networks. This is fine for the same-room use case.

4. **Firefox ICE gathering**: Firefox may not fire `icegatheringstatechange` reliably. Use a polling fallback that checks `iceGatheringState` every 500ms and proceeds after 15 seconds regardless.

5. **BarcodeDetector binary data**: Some implementations of `BarcodeDetector` may mangle binary QR data as they attempt to decode it as UTF-8. Test this explicitly. If mangling occurs, encode the binary packet as base45 or base64 before QR encoding (with a slight payload size increase) and decode on the other side.
